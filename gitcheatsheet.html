Github Cheatsheet

Méthodes de travail collaboratives sur Github

Feature Branch

Feature Branch (branche par fonctionnalité) est une pratique de développement flexible permettant à un développeur de travailler sur plusieurs branches simultanément - et même à plusieurs développeurs de travailler sur la même branche (évite aussi des confusions du type « pourquoi a commit sur la branche de b ?»). Ce qui facilite aussi la gestion des pull requests car une branche appelée add-button-to-home-page est plus explicite qu’une branche appelée bob.

Developer Branch

Une autre pratique qui consiste à créer une branche par développeur, sur laquelle vous pousserez votre code pour les différentes pages/fonctionnalités sur lesquelles vous travaillez. A l’avantage d’être simple d’utilisation car ne nécessite pas de changer de branche régulièrement.

Issues

Une issue peut être considérée comme une tâche à effectuer ultérieurement. Une issue contient, un titre, une description, vous pouvez également y ajouter des liens et des images. Dans le cas du développement collaboratif les issues sont utiles pour préparer les fonctionnalités à implémenter ou des bugs à fix. Une issue peut être assignée à un ou plusieurs développeurs. Dans un message de commit vous pouvez faire référence à une issue en précisant son id. 

Message faisant référence à l'issue #10 :git commit -m "#10 message"

De plus, lorsque vous utilisez certains mots clés dans vos messages de commit contenant également l'id de l'issue, vous pouvez effectuer certaines actions comme fermer une issue en cours. 

Ce message va automatiquement fermer l'issue #10 :git commit -m "Fixes #10"

Pull Requests

Une fois que vous estimez avoir fini votre travail, il faut alors l’ajouter à la branche principale master qui contient le code fonctionnel de l’application. Pour éviter les bugs, avant d’ajouter du code au master il faut ouvrir une pull request. Une comparaison sera faite des différents fichiers sur le master à ceux que vous avez modifiés/ajoutés/supprimés. Le code est ensuite vérifié par un (ou plusieurs) autre développeur qui donnera son feu vert pour l’ajouter au master.

Initialization

Initialise un dossier local en tant que dossier git :git init
Récupérer en local un repository git via une URL. :git clone url

Pousser son code

Ces étapes sont à exécuter dans cet ordre pour pousser votre code sur le repository distant.

Ajouter tous les fichiers au prochain commit :git add .

ou

Ajouter uniquement les fichiers modifiés au prochain commit :git add -u
Commit les derniers changements trackés :git commit -m "message descriptif"
Publier les changements locaux sur le repository distant :git push

Branches

Les branches permettent de coder de votre côté sans perturber le master qui est la branche principale du projet, elle ne doit contenir que du code fonctionnel. En fonction de la méthode choisie, vous aurez plus ou moins souvent besoin de créer un changer fréquemment de branche. 

Liste les branches existantes :git branch
Se placer sur une branche existante :git checkout nom-de-la-branche
Créer une nouvelle branche et se placer dessus :git checkout -b nom-de-la-branche

Dans le cas où vous avez besoin de changer régulièrement de branche, vous aurez parfois des modifications en cours sur la branche actuelle qui n’ont pas été commit, ce qui vous empêchera de changer de branche. Dans ce cas, vous pouvez stocker vos modifications temporairement dans un stash. 

Stocker les changements de la branche actuelle dans un stash :git stash
Afficher la liste des stash disponibles :git stash list
Récupérer et appliquer les données du dernier stash à la branche actuelle, 

vous pouvez également préciser l’id ou le nom du stash (ex: git stash pop stash@{3}) :git stash pop
Afficher les données d’un stash :git stash show stash@{0}
Merge Branches

Une fois votre travail terminé sur votre branche et la Pull Request validé, vous aurez besoin d'importer le code du master (ou d'une autre branche) mis à jour sur votre branche en local. 

Pour importer en local le code d'une branche distante spécifique 
("origin" fait référence au repo distant à partir duquel un projet a été cloné) :git pull origin branche

Il existe aussi des options avec des interfaces graphiques (ex: GitKraken) qui facilitent le merge de code entre différentes branches et la gestion de conflits éventuels. A privilégier si vous n'êtes pas à l'aise avec git ou le terminal.
